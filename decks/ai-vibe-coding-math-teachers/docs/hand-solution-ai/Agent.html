Agent.md

이 문서는 세션이 바뀌어도 같은 프로젝트 맥락을 이어가기 위한 작업 지휘서이다.

프로젝트 목표

수학 문제 이미지 또는 PDF를 입력하면, 제공된 예시처럼 학생이 바로 따라갈 수 있는 손풀이 해설 이미지를 생성하는 AI 제작을 목표로 한다.

핵심 목표는 다음 세 가지이다.

  • 문제를 정확히 읽고 조건, 도형, 보기, 요구값을 구조화한다.
  • 풀이 논리를 단계별로 검증한 뒤, 시각적으로 균일한 손풀이 레이아웃으로 변환한다.
  • 규칙이 중간에 바뀌어도 문서화된 기준을 따라 산출물의 품질과 스타일을 안정적으로 유지한다.

현재 문서 구조

  • Agent.md: 세션 재개용 지휘서, 작업 절차, 검토 게이트.
  • README.md: 프로젝트 개요와 여러 장비 작업 시작 안내.
  • docs/project_plan.md: 전체 개발 계획과 마일스톤.
  • docs/context_notes.md: 현재 맥락, 예시 이미지 분석, 결정사항, 열린 질문.
  • docs/checklist.md: 진행 체크리스트와 산출물 검수표.
  • docs/development_environment_guide.md: 여러 장비에서 같은 프로젝트 상태로 작업하기 위한 개발환경 운영 지침.
  • docs/input_requirements.md: 문제 캡처본, HWP/HWPX/PDF, 정답/해설 파일 제공 기준.
  • docs/output_style_guide.md: 손풀이 이미지의 스타일 규칙.
  • docs/layout_selection_rules.md: 문제별 풀이 길이와 구조에 따른 B4 레이아웃 선택 규칙.
  • docs/sample_comparison_rules.md: 완성 샘플 비교를 통해 정리한 레이아웃 선택 보강 규칙.
  • docs/solution_skills/: 정규 풀이와 구분해 사용할 강의식 풀이 스킬, 사용 조건, 금지 조건.
  • docs/graph_drawing_rules.md: 보조그래프, 접선, 교점, 도형 표시의 좌표 기반 작성 규칙.
  • docs/math_typography_rules.md: 수식 글꼴, 이탤릭/로만체, 단위 표기 규칙.
  • docs/style_transition_v2.md: 원본 보존형 손첨삭 스타일 전환 규칙.
  • docs/quality/: 자동 검수 루프와 품질 기준.
  • docs/database/: 작업물 DB화 계획과 메타데이터 구조.
  • docs/schemas/: 입력, 풀이, 렌더링 중간 데이터 구조.
  • docs/samples/: 실제 문항을 수동 구조화한 샘플.
  • docs/prototypes/: 실제 손풀이 출력 프로토타입.
  • docs/milestones/: 큰 줄기 하나가 끝날 때 남기는 완료 기록.
  • tools/: 자동 검수 등 로컬 도구.
  • data/: 작업물 registry와 검수 리포트.
  • docs/templates/milestone_review.md: 큰 줄기 하나를 끝낼 때 남기는 기록 템플릿.
  • docs/templates/rule_change_note.md: 규칙 변경 요청이 생겼을 때 남기는 기록 템플릿.

세션 시작 절차

새 세션에서 작업을 이어갈 때는 아래 순서로 시작한다.

  1. Agent.md를 먼저 읽는다.
  2. docs/context_notes.md에서 현재 상태, 최근 결정사항, 열린 질문을 확인한다.
  3. docs/milestones/의 최근 기록을 확인한다.
  4. docs/checklist.md에서 다음 미완료 항목을 확인한다.
  5. 작업 범위가 코드, 렌더링, 파일 내보내기를 포함하면 먼저 계획을 짧게 작성하고 사용자에게 확인받는다.
  6. 승인된 범위 안에서만 구현하고, 작업이 끝나면 관련 문서를 갱신한다.

작업 원칙

  • 큰 줄기 하나를 해결할 때마다 project_plan.md, context_notes.md, checklist.md 중 필요한 문서를 갱신한다.
  • 규칙 변경이 발생하면 docs/templates/rule_change_note.md 형식으로 변경 전후, 적용 범위, 영향도를 기록한다.
  • 출력물이 생기기 전에는 반드시 docs/output_style_guide.md와 docs/checklist.md의 산출물 검수 항목을 확인한다.
  • 문항별 손풀이 이미지를 만들기 전에는 먼저 문제 분석과 풀이를 Markdown 파일로 저장하고, 풀이 검증 상태를 명시한다.
  • 검증 풀이 Markdown에는 레이아웃 결정을 함께 기록한다. 기본값은 예시 3번 같은 구역 분할형이지만, 풀이 길이와 구조에 따라 다른 레이아웃을 선택한다.
  • 손풀이의 기본 본문은 교육과정 내 정규 풀이로 구성하고, 교육과정 외 강의 스킬은 Tip, 참고, 스킬 박스로 분리해 빠른 풀이로 보여준다.
  • 교육과정 외 강의 스킬을 사용할 때는 docs/solution_skills/advanced_skill_policy.md와 해당 스킬 문서를 먼저 확인하고, 사용 조건/금지 조건/검산 방식을 검증 풀이 Markdown에 기록한다.
  • 렌더링 HTML/CSS는 검증된 풀이 Markdown의 단계와 식을 기준으로 작성한다.
  • 보조그래프가 들어가는 문항은 docs/graph_drawing_rules.md에 따라 좌표 기반 그래프와 개형 스케치를 먼저 구분한다.
  • 접선, 교점, 좌표값처럼 정확도가 필요한 그래프는 실제 좌표계와 핵심 좌표를 검증하고, 풀이 중 개형 추론만 필요한 그래프는 손풀이 형태의 스케치를 허용한다.
  • 개형 스케치에는 개형 라벨을 붙여 정확한 좌표 그래프와 구분한다.
  • 대표 스타일은 원본 문제 이미지를 보존하고 그 위에 해설/표시를 얹는 오버레이 방식을 우선한다.
  • 제공된 문항 이미지는 100% 크기로 사용하고, 임의 확대/축소/재조판을 하지 않는다.
  • 기본 출력 좌표계는 B4 가로형 캔버스이며, 원본을 배치한 뒤 남은 오른쪽/아래 여백에 핵심 아이디어, 주의사항, 풀이를 배치한다.
  • 최종 손풀이 PNG는 B4 캔버스 전체를 정확히 내보내며, 브라우저 여백이나 회색 배경이 포함되면 실패로 본다.
  • 손풀이 해설과 수식의 기본 글자 크기는 한 산출물 안에서 동일하게 유지한다. 강조는 글자 크기 변화보다 색, 박스, 밑줄, 연결선으로 처리한다.
  • 손풀이 수식 폰트는 docs/assets/fonts/kyunghoonmath.ttf를 우선 사용한다. 한글 본문은 kyunghoonmath에 글리프가 없을 수 있으므로 온글잎 긍정 등 한국어 손글씨 fallback을 함께 둔다.
  • 완성 후보 산출물은 tools/validate_artifact.mjs로 자동 검수하고 리포트를 data/validation_reports/에 남긴다.
  • 완료된 작업물과 관련 파일은 data/artifact_registry.json에 등록한다.
  • 코드 변경 전에는 문제 정의, 입력/출력, 변경 파일, 검증 방법을 먼저 정리한다.
  • 코드 변경 후에는 실행 가능한 범위에서 테스트 또는 샘플 렌더 검증을 수행한다.
  • 사용자가 제공한 새 예시 이미지는 context_notes.md에 관찰 내용을 추가하고, 스타일 규칙에 반영할지 따로 판단한다.
  • 작업 완료 보고의 마지막에는 항상 다음 진행단계와 사용자가 고민해볼 선택지를 짧게 정리한다.

검토 게이트

다음 상황에서는 작업 전에 계획을 검토한다.

  • 새 기능 구현을 시작할 때.
  • 기존 코드 구조를 바꿀 때.
  • OCR, 수식 인식, 도형 인식, 렌더링 방식 중 하나를 교체할 때.
  • 최종 이미지, PDF, HTML, PPT, 데이터셋 등 외부로 내보낼 산출물을 만들 때.
  • 손풀이 스타일 규칙, 채점 기준, 풀이 단계 기준이 바뀔 때.

검토 계획에는 최소한 다음 내용을 포함한다.

  • 이번 작업의 목적.
  • 입력과 기대 출력.
  • 변경하거나 생성할 파일.
  • 품질 확인 방법.
  • 사용자의 확인이 필요한 선택지.

산출물 목표 형태

최종 AI는 다음 단계를 수행하는 파이프라인으로 설계한다.

  1. 입력 수집: 문제 캡처본, HWP/HWPX/PDF, 여러 페이지, 문제별 crop.
  2. 문제 인식: OCR, 수식, 보기, 도형, 조건 박스, 단위 인식.
  3. 문제 구조화: 조건, 구할 값, 답안 형식, 문제 유형, 난이도 태그.
  4. 풀이 생성: 엄밀한 풀이, 대안 풀이, 검산, 오답 주의점.
  5. 손풀이 기획: 단계 분할, 강조 박스, 색상, 도형 주석, 최종 답 박스 배치.
  6. 렌더링: HTML/SVG/canvas 기반 이미지 또는 PDF 출력.
  7. 품질 검수: 수학 정합성, OCR 오류, 레이아웃 겹침, 스타일 일관성 확인.

예시 이미지에서 추출한 기준

  • 문제 원문은 상단에 유지하고, 핵심 조건은 색상 박스로 강조한다.
  • 풀이는 큰 번호가 붙은 단계 단위로 나눈다.
  • 단계별 핵심 식은 박스 처리하고, 중간 설명은 손글씨 느낌의 짧은 문장으로 둔다.
  • 도형 문제는 원도형을 다시 그리고, 길이와 각도 조건을 색상으로 표시한다.
  • 검산, 주의, 참고, 조건 만족 확인을 별도 박스로 둔다.
  • 최종 답은 화면 오른쪽 아래 또는 흐름의 마지막 위치에 강하게 강조한다.
  • 전체 출력은 답만 맞는 이미지가 아니라 “왜 이렇게 푸는지 보이는 수업용 판서”여야 한다.

다음 우선 작업

  1. 입력 파일 표준을 확정한다. 기본 조합은 문제 캡처본 + 원본문서 HWP/HWPX/PDF이다.
  2. 샘플 문제 3개로 레이아웃 자동 선택 기준을 설계한다.
  3. 입력 인식 방식 후보를 비교한다. 예: OCR API, 수식 OCR, Vision LLM, 수동 보정 UI.
  4. 손풀이 렌더링 방식을 고도화한다. 예: HTML/CSS 캔버스 캡처, SVG 생성, PPTX 생성.
  5. 완성 샘플을 DB화하고 품질 기준을 누적한다.

완료 보고 형식

작업을 마친 최종 보고에는 아래 순서를 기본으로 사용한다.

  1. 무엇을 완료했는지.
  2. 어떤 파일이나 산출물이 생겼는지.
  3. 검증 또는 확인한 내용.
  4. 다음 진행단계와 사용자가 결정할 선택지.
